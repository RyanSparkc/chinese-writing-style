## 一、句子結構原則

### 規則 1：使用主動語態

主動語態比被動語態更直接有力。中文的「被」字句應謹慎使用。

| 被動（避免） | 主動（建議） |
|------------|------------|
| 檔案已被刪除 | 系統已刪除檔案 |
| 此功能將被棄用 | 我們將棄用此功能 |
| 錯誤訊息會被顯示 | 系統會顯示錯誤訊息 |
| 資料被成功儲存 | 資料儲存成功 |
| 請求被伺服器拒絕 | 伺服器拒絕請求 |

**例外：** 當動作執行者不重要或刻意隱藏時，可使用被動語態：

- 他被解僱了。（強調結果，執行者不重要）
- 這棟建築被列為古蹟。（執行者為政府，已知）

### 規則 2：用肯定句式

肯定句比否定句更清楚直接。避免雙重否定。

| 否定（避免） | 肯定（建議） |
|------------|------------|
| 不是不可以 | 可以 |
| 並非沒有問題 | 有問題 |
| 不無道理 | 有道理 |
| 不是很常見 | 少見 |
| 沒有不同意 | 同意 |

**否定句的正確用法：** 用於強調對比時

- 這不是 bug，是 feature。
- 重要的不是結果，而是過程。

### 規則 3：具體明確

避免空泛抽象的詞彙，使用具體明確的描述。

| 抽象（避免） | 具體（建議） |
|------------|------------|
| 效能有所提升 | 載入時間從 3 秒降至 0.5 秒 |
| 近期會完成 | 預計週五前完成 |
| 大量資料 | 超過 10 萬筆記錄 |
| 系統出了一些問題 | 資料庫連線逾時 |
| 需要一些時間 | 約需 2 小時 |

### 規則 4：刪除贅字

精簡的文字更有力。每個字都應該有存在的意義。

| 冗贅（避免） | 精簡（建議） |
|------------|------------|
| 進行討論 | 討論 |
| 進行分析 | 分析 |
| 進行測試 | 測試 |
| 作出決定 | 決定 |
| 作出回應 | 回應 |
| 給予支援 | 支援 |
| 加以改善 | 改善 |
| 予以刪除 | 刪除 |
| 達到目的 | 達成目的 |
| 造成影響 | 影響 |

#### 常見贅詞一覽

以下詞彙通常可以刪除：

| 贅詞 | 說明 |
|-----|-----|
| 的話 | 「如果有問題的話」→「如果有問題」 |
| 然後 | 常被濫用為口語連接詞 |
| 基本上 | 「基本上沒問題」→「沒問題」 |
| 其實 | 「其實很簡單」→「很簡單」 |
| 所謂的 | 「所謂的最佳實踐」→「最佳實踐」 |
| 相關的 | 「相關的設定」→「設定」 |
| 一定的 | 「有一定的風險」→「有風險」 |
| 有關 | 「有關設定的說明」→「設定說明」 |

### 規則 5：相關詞語放一起

主語和謂語、動詞和賓語之間不要插入太長的修飾語。

| 分離（避免） | 聚合（建議） |
|------------|------------|
| 使用者在完成驗證並同意條款後可以存取資料 | 使用者完成驗證並同意條款後，可存取資料 |
| 這個由多位資深工程師共同開發的功能 | 多位資深工程師共同開發的這個功能 |

**原則：** 讓讀者能夠快速找到句子的主幹（誰做了什麼）。

### 規則 6：重點放句尾

句子最後的詞語最容易被記住。把重要資訊放在句尾。

| 重點在前（較弱） | 重點在後（較強） |
|---------------|---------------|
| 效能是這次更新的重點 | 這次更新的重點是效能 |
| 資料完整性必須確保 | 必須確保資料完整性 |
| 三個錯誤被找到了 | 我們找到了三個錯誤 |

### 規則 7：一段一主題

每個段落應該只討論一個主題，以主題句開頭。

**不好的例子：**
> 這個 API 支援 JSON 和 XML 格式。認證方式採用 OAuth 2.0。回應時間通常在 100 毫秒內。我們建議使用快取來提升效能。

**好的例子：**
> **資料格式**
> 這個 API 支援 JSON 和 XML 兩種格式。預設為 JSON，可透過 `Accept` 標頭指定格式。
>
> **認證方式**
> 所有請求需使用 OAuth 2.0 認證。請在標頭中附上 Bearer Token。
>
> **效能考量**
> API 回應時間通常在 100 毫秒內。建議使用快取機制以提升效能。

### 規則 8：保持一致的時態

在同一份文件中，保持時態一致。

| 不一致（避免） | 一致（建議） |
|-------------|------------|
| 系統會檢查權限，然後回傳了結果 | 系統會檢查權限，然後回傳結果 |
| 按下按鈕後，視窗關閉了 | 按下按鈕後，視窗會關閉 |

## 二、標點符號規則

### 規則 9：使用全形標點

中文內容使用全形標點符號。

| 標點 | 全形（正確） | 半形（錯誤） |
|-----|------------|------------|
| 逗號 | ， | , |
| 句號 | 。 | . |
| 頓號 | 、 | 無 |
| 分號 | ； | ; |
| 冒號 | ： | : |
| 問號 | ？ | ? |
| 驚嘆號 | ！ | ! |
| 引號 | 「」『』 | "" '' |
| 括號 | （）【】 | () [] |
| 書名號 | 《》〈〉 | 無 |

### 規則 10：引號使用

台灣繁體中文使用「」和『』作為引號。

- 第一層引號：「」
- 引號內再引用：『』

**範例：**
> 他說：「我讀過《論語》，其中『學而時習之』這句話讓我印象深刻。」

**注意：** 程式碼、指令、檔案名稱使用反引號 \`code\`，不使用中文引號。

### 規則 11：頓號與逗號

- **頓號（、）：** 用於並列詞語之間
- **逗號（，）：** 用於句子內部停頓

| 情境 | 正確用法 |
|-----|---------|
| 並列名詞 | 蘋果、橘子、香蕉 |
| 並列形容詞 | 快速、簡單、可靠 |
| 並列子句 | 按下按鈕，輸入密碼，確認送出 |

**注意：** 三個以上並列項目，最後一項前不加「和」或「與」。

| 錯誤 | 正確 |
|-----|-----|
| 紅色、藍色、和綠色 | 紅色、藍色、綠色 |

### 規則 12：冒號使用

冒號後面通常接完整句子或列表，不接單詞。

| 錯誤 | 正確 |
|-----|-----|
| 狀態：成功 | 狀態：操作成功 |
| 注意：危險 | 注意：此操作有風險 |

**例外：** 表格、表單中的標籤可以使用「標籤：值」格式。

### 規則 13：避免連續相同標點

不要連續使用相同的標點符號。

| 錯誤 | 正確 |
|-----|-----|
| 真的嗎？？？ | 真的嗎？ |
| 太棒了！！！ | 太棒了！ |
| 嗯...... | 嗯…… |

**刪節號規則：** 使用「……」（兩個全形刪節號，共六點），不使用「...」（三個半形句點）。

## 三、常見贅字與誤用

### 「的」的濫用

「的」是最常被濫用的字。很多時候可以省略。

| 冗贅 | 精簡 |
|-----|-----|
| 我的想法是 | 我想 |
| 系統的效能 | 系統效能 |
| 這個的原因是 | 原因是 |
| 成功的機率 | 成功機率 |
| 使用者的資料 | 使用者資料 |

**判斷原則：** 刪除「的」後如果意思不變，就應該刪除。

### 「進行」的濫用

「進行」幾乎總是多餘的。

| 冗贅 | 精簡 |
|-----|-----|
| 進行測試 | 測試 |
| 進行修改 | 修改 |
| 進行討論 | 討論 |
| 進行分析 | 分析 |
| 進行驗證 | 驗證 |
| 進行更新 | 更新 |

### 「作出/做出」的濫用

| 冗贅 | 精簡 |
|-----|-----|
| 作出決定 | 決定 |
| 作出回應 | 回應 |
| 作出調整 | 調整 |
| 做出選擇 | 選擇 |
| 做出貢獻 | 貢獻 |

### 其他常見贅詞

| 贅詞 | 精簡後 | 範例 |
|-----|-------|-----|
| 目前現在 | 目前/現在 | 目前現在的狀態 → 目前狀態 |
| 可以能夠 | 可以/能 | 可以能夠使用 → 可使用 |
| 必須要 | 必須/要 | 必須要完成 → 必須完成 |
| 已經...了 | 已.../...了 | 已經完成了 → 已完成 |
| 正在...中 | 正在.../...中 | 正在處理中 → 處理中 |
| 將會 | 將/會 | 將會發布 → 將發布 |
| 並且 | 並/且 | 儲存並且關閉 → 儲存並關閉 |

### 常見誤用詞彙

| 誤用 | 正確 | 說明 |
|-----|-----|-----|
| 權限 vs 權利 | 權限（系統）、權利（法律） | 使用者權限，公民權利 |
| 帳號 vs 賬號 | 帳號（台灣） | 台灣用「帳」，中國用「賬」 |
| 軟體 vs 軟件 | 軟體（台灣） | 台灣用「軟體」 |
| 視窗 vs 窗口 | 視窗（台灣） | 台灣用「視窗」 |
| 滑鼠 vs 鼠標 | 滑鼠（台灣） | 台灣用「滑鼠」 |
| 程式 vs 程序 | 程式（代碼）、程序（流程） | 注意區分 |
| 預設 vs 默認 | 預設（台灣） | 台灣用「預設」 |
| 執行 vs 運行 | 執行（台灣） | 台灣用「執行」 |

## 四、中英混排規則

### 規則 14：中英文之間加空格

中文與英文、數字之間應加入一個半形空格。

| 錯誤 | 正確 |
|-----|-----|
| 使用React開發 | 使用 React 開發 |
| 版本號是1.0.0 | 版本號是 1.0.0 |
| 請參考API文件 | 請參考 API 文件 |

### 規則 15：專有名詞保留原文

技術專有名詞保留英文原文，不強行翻譯。

| 保留原文 | 避免翻譯 |
|---------|---------|
| API | 應用程式介面 |
| Git | 版本控制系統 |
| React | 反應 |
| Docker | 碼頭工人 |
| bug | 臭蟲 |
| commit | 提交（可用，但 commit 更精確） |

**可翻譯的詞彙：** 已有廣泛使用的中文對應詞

| 英文 | 中文 |
|-----|-----|
| file | 檔案 |
| folder/directory | 資料夾 |
| button | 按鈕 |
| database | 資料庫 |
| server | 伺服器 |
| user | 使用者 |

### 規則 16：全形標點與英文

全形標點符號與英文之間**不加空格**。

| 錯誤 | 正確 |
|-----|-----|
| 請使用 React 。 | 請使用 React。 |
| 什麼是 API ？ | 什麼是 API？ |
| 例如 ：Node.js | 例如：Node.js |

### 規則 17：括號使用

- **中文內容：** 使用全形括號（）
- **純英文/程式碼：** 使用半形括號 ()
- **括號內為英文：** 前後不加空格

| 情境 | 正確用法 |
|-----|---------|
| 中文說明 | 請參考設定檔（見下方） |
| 英文縮寫 | 應用程式介面（API） |
| 純英文 | React (version 18) |

### 規則 18：連結文字

連結文字應該有意義，避免「點擊這裡」。

| 錯誤 | 正確 |
|-----|-----|
| 點擊這裡查看文件 | 查看 [API 文件](url) |
| 更多資訊請點此 | 參考 [安裝指南](url) |

## 五、技術文件寫作

### 規則 19：使用第二人稱

技術文件中，對讀者使用「你」而非「使用者」。

| 疏遠 | 親切 |
|-----|-----|
| 使用者需要先安裝 Node.js | 你需要先安裝 Node.js |
| 使用者可以透過以下方式設定 | 你可以透過以下方式設定 |

### 規則 20：使用祈使句

指示步驟時使用祈使句（省略主詞）。

| 冗長 | 簡潔 |
|-----|-----|
| 你需要點擊「確定」按鈕 | 點擊「確定」 |
| 使用者應該輸入密碼 | 輸入密碼 |
| 請你選擇檔案 | 選擇檔案 |

### 規則 21：程式碼格式

- 行內程式碼使用反引號：\`code\`
- 程式碼區塊使用三個反引號並標註語言
- 檔案名稱、指令、變數名稱都用反引號

**範例：**
> 執行 \`npm install\` 安裝相依套件。設定檔位於 \`config.json\`。

### 規則 22：錯誤訊息寫法

錯誤訊息應該：
1. 說明發生什麼問題
2. 說明可能的原因
3. 提供解決方法

| 不好 | 好 |
|-----|-----|
| 錯誤 | 無法連線到資料庫：連線逾時。請檢查網路設定或稍後重試。 |
| 操作失敗 | 檔案上傳失敗：檔案大小超過 10 MB 限制。請壓縮檔案後重新上傳。 |

### 規則 23：版本說明寫法

版本說明應該清楚說明改動內容。

**不好的例子：**
> - 修復一些問題
> - 改進效能
> - 更新相依套件

**好的例子：**
> - 修復：登入頁面在 Safari 無法顯示驗證碼
> - 改進：首頁載入時間從 3 秒降至 0.8 秒
> - 更新：React 從 17.0.2 升級至 18.2.0

## 六、數字與單位

### 規則 24：阿拉伯數字與中文數字

| 使用阿拉伯數字 | 使用中文數字 |
|--------------|------------|
| 精確數值：100 MB | 約數：幾百人 |
| 日期：2024 年 1 月 | 成語：一目十行 |
| 版本：v2.0.0 | 序數詞：第一次 |
| 金額：$500 | 慣用語：三思而行 |

### 規則 25：數字與中文之間加空格

| 錯誤 | 正確 |
|-----|-----|
| 共有100位使用者 | 共有 100 位使用者 |
| 檔案大小為5MB | 檔案大小為 5 MB |
| 版本1.0.0 | 版本 1.0.0 |

### 規則 26：單位寫法

- 單位與數字之間加空格
- 常見單位使用英文縮寫

| 錯誤 | 正確 |
|-----|-----|
| 10MB | 10 MB |
| 5秒 | 5 秒 |
| 100% | 100%（百分號例外，不加空格） |

## 快速檢查清單

撰寫完成後，用以下清單檢查：

- [ ] 是否有不必要的「的」、「進行」、「作出」？
- [ ] 是否使用主動語態？
- [ ] 中英文之間是否有空格？
- [ ] 數字與中文之間是否有空格？
- [ ] 標點符號是否使用全形？
- [ ] 專有名詞是否保持一致？
- [ ] 句子是否簡潔有力？
- [ ] 段落是否只討論一個主題？
- [ ] 重要資訊是否放在句尾？
- [ ] 錯誤訊息是否提供解決方法？
